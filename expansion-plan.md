## A. 小米的隱藏感情線解鎖條件（取代舊設定）

小米是同事，她的感情線是**全遊戲唯一的隱藏線**，不能像其他四位一樣 Lv4 點頭像就開。設計如下，五個條件**全部滿足**後才觸發：

| # | 條件 | 儲存欄位 | 說明 |
|---|---|---|---|
| 1 | Lv ≥ 4 | `s.lv` | 與其他人相同的基本門檻 |
| 2 | 讀完小米的全部同事閒聊（6 章） | `s.chat.komi >= 6` | 含「以毒攻毒」協助任務（`s.helpDone.komi`） |
| 3 | 累計找她淨化心魔 ≥ 10 次 | `s.komiCare`（新增計數器） | 在 talkTo 的 komi 淨化路徑 `+1` |
| 4 | 觸發當下「單身」 | 所有 `s.rel[*].dating` 為 false | **她不當第三者**——正在交往中永遠不會觸發，這是隱藏感的核心 |
| 5 | 剛做完「加班」後進入茶水間點她 | `s.lastActionOvertime`（新增旗標，work(true) 設 true、其他行動設 false） | 對應她 intro 的場景（加班後的深夜茶水間，她在放空） |

**引擎掛鉤（index.html）**：
1. komi 淨化路徑（搜 `淨化完成`）加 `s.komiCare=(s.komiCare||0)+1;`
2. `work()` 內 `if(overtime) s.lastActionOvertime=true;`；其餘 action 開頭統一設 false（最簡做法：在 `pay()` 成功回呼前設 false，work 再覆寫）。
3. `talkTo("komi")` 的隨機閒聊 fallback 之前插入條件檢查：全滿足 → `meetHeroine("komi")` 並 return。
4. 提示線索（讓玩家找得到隱藏線）：
   - 彩花閒聊池加一句：「小米最近打烊後常一個人在茶水間發呆呢……有人陪她說說話就好了。」
   - 阿低閒聊池加一句：「小米的下班打卡紀錄最近都很晚。系統看得到，但系統不會安慰人。」
   - 兩句在 `s.chat.komi>=6` 之後才進池（避免太早暴雷）——在 SMALLTALK 選線時加條件即可。
5. 解鎖瞬間的特殊感：intro 播完後 log「【隱藏感情線】你注意到了那個總是在傾聽別人的人。」

## B. 角色出沒／漫遊系統（世界活起來的關鍵）

### 設計原則
- 每個角色有「本座位」與「偶爾出沒地」。玩家**移動到某地點時擲骰**，決定該地這次有哪些「訪客」；離開後重擲。
- 訪客顯示在 npc-row，名字帶「（訪）」字樣；點擊播放**專屬的出沒對話**（每組 3～5 段輪替），部分給小獎勵。
- 女主角也會**不在家**：每次進她的地點 10% 機率「今日公休」（門上字條，各寫一句有味道的留言），此時她可能正在別處漫遊——在別處遇到她算「偶遇」，好感 +3（比平常約會少但驚喜感強）。

### 引擎掛鉤（index.html，約 40 行）
1. story.js 新區塊 `ROAM:[ {id, at, chance, pool:[[lines],[lines],...]} ]`、`ABSENT_NOTES:{alin:"...", fei:"...", jixu:"..."}`（公休字條）。
2. `move()` 成功後：`rollRoam()` —— 清空 `s.roamNow`，對 ROAM 表中 `at===s.loc` 的條目擲骰，命中者塞進 `s.roamNow`（陣列，不存檔也可，用記憶體變數即可）。
3. `render()` 的 npc-row：residents 之後補畫 `roamNow` 的訪客（名字加「（訪）」、邊框用灰金色）；女主角公休判定：進駐地時 10% 設 `absent` 旗標並在 npc-row 顯示「（公休）」字條卡。
4. 點訪客 → `say(pool 隨機一段)`；特殊效果（封口費、komiCare、偶遇好感）在對話結尾的函式型 text 內處理（參照現有 villain 事件寫法）。
5. 老闆彩蛋：點擊後 `s.files.boss=Math.max(s.files.boss||0,3)` 並 log 名冊更新。

### 字數配額
12 組 × 平均 4 段 × 每段 ~90 字 ≈ **4,300 字**（計入公司線）。公休字條 ×5 ≈ 200 字（計入感情線）。

## C. 手機／LINE 系統（晴晴的主要互動方式）

### 設計原則
晴晴是網紅，**本人極難遇到**：初次滿版登場後，她在頂樓/工位實體出現的機率降為 20%（拍攝行程滿檔）。主要互動改為**手機**——這也符合「網紅的戀愛就是活在訊息裡」的主題。

### UI 規格（新 modal：#phone-overlay，仿聊天 App 深色風）
- header 新增「手機」按鈕（介紹完成後出現）。
- 兩個分頁：
  - **動態牆**：晴晴的貼文流（POSTS 資料池，每日刷 1-2 則）。每則可「按讚」（每日首讚 fav+1）。交往後她的貼文內容變化（見 romance-plan 的 AFTERMATH 也走這裡）。
  - **聊天室**：LINE 式對話串。她依好感度階段主動傳訊（THREADS 資料，見下）；你用 2-3 個**選項按鈕**回覆，不同回覆好感增幅不同（+1~+3，選錯只是好笑不懲罰）。已讀梗必備（「已讀 3 分鐘……她是不是在拍片」）。
- 交往後追加：晚安訊息（每日一則，池 ≥15 則）、突襲自拍（用她的三段表情圖當「照片」）、「語音通話」純文字演出。

### 資料格式（romance.js 新區塊）
```js
PHONE:{
  posts:[ {text:"貼文內容", fav:1}, ... ],                       // ≥25 則，交往前後混用
  threads:[ { need:{fav:10}, id:"t1", msgs:[
      {who:"liang", text:"欸，今天頂樓的雲超好看，可惜你在上班 😏"},
      {who:"choice", opts:[
        {label:"拍給我看！", fav:2, reply:"馬上傳！（一張構圖完美的雲）這張只給你看喔"},
        {label:"我翹班上去", fav:3, reply:"不行啦笨蛋！！……五分鐘，我等你（笑）"},
        {label:"已讀", fav:0, reply:"？？？你敢已讀我？全網三十萬人想收到這則訊息欸！"}
      ]},
    ]}, ... ],                                                    // ≥12 串，依 fav 10/25/45/70/90/110… 展開
  goodnight:[ "晚安訊息池", ... ],                                 // ≥15 則（交往後）
  absent_line:"（拍攝中，晚點回你）"                                // 她已讀不回時的狀態
}
```
threads 建議劇情軸：日常閒聊 → 抱怨業配 → 深夜脆弱（黑粉/掉粉時期，銜接 DEEP 60 章）→ 曖昧試探 → 告白前的鋪陳（fav 110 的 thread 結尾她約你上頂樓 = 導向告白）。

### 引擎掛鉤（index.html，約 90 行）
1. phone modal：仿 roster 的 overlay 結構；兩分頁用兩個 div 切換（不用做即時聊天，開啟時渲染即可）。
2. 每日刷新：`s.phoneDay`（toDateString）變更時：解鎖當日貼文 index、重置按讚旗標、交往中推播晚安訊息。
3. threads 進度：`s.phoneThread`（已完成的 thread id 集合）；開啟聊天室時找第一個 `fav 達標且未完成` 的 thread 逐句播放，choice 用按鈕。
4. 晴晴實體出沒改 20%：她原本掛在 ROM_LOC 的頂樓/工位——出現與否在 render npc-row 時擲骰（用當日 seed 固定，避免每秒閃爍：`seed = 日期字串+loc` 簡單 hash）。
5. 其他四位女角的手機（**第二階段，字數若有餘再做**）：交往後每人 8 則 LINE 訊息池，共用聊天室 UI。

### 字數配額
posts 25×30 + threads 12×180 + goodnight 15×30 + 雜項 ≈ **3,600 字**（計入感情線 50k 內，替換 romance-plan 配額表中 COURTSHIP 的 1,000 字與 OUTINGS 的 600 字，其餘從 EVENTS 池挪——執筆時以總量 50k 為準自行微調即可）。
